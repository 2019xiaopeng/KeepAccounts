# KeepAccounts AI 人性化交互设计（V1）

> 版本：2026-03-29
> 目标：让核心操作更符合直觉，降低记账门槛，并让 AI 的帮助“有用但不打扰”。

## 1. 设计目标

- 低认知成本：用户在不学习教程的情况下可完成首笔记账。
- 一次说清优先：尽量在单轮对话内完成记账，不强迫多轮补信息。
- 纠错成本低：发现错误后可在当前上下文一键改正。
- 建议有边界：AI 给建议要可解释、可关闭、可降频。
- 保持温和一致：语气稳定、简短、不过度拟人和说教。

## 2. 交互原则

### 2.1 最小打扰原则
- 只在关键信息缺失时提问。
- 每次最多提 1 个问题，优先给出候选答案。
- 示例：
  - 用户：午饭花了点钱
  - AI：按餐饮 35 元记账可以吗？

### 2.2 回执即操作原则
- 记账回执卡片内直接提供：修改金额、修改分类、修改时间、撤销。
- 不强制跳转到二级页面。

### 2.3 可解释原则
- AI 给出建议时附带简单依据（例如“近 7 天餐饮支出比上周高 28%”）。
- 禁止出现无法追溯来源的“主观判断型提醒”。

### 2.4 用户掌控原则
- 所有建议功能都提供开关和强度选项（关闭/轻度/标准）。
- 用户可随时关闭某类提醒，不影响记账主流程。

## 3. 核心场景设计

### 3.1 一句话快记
- 输入："咖啡 22"
- 系统动作：
  1. AI 解析金额、分类、时间。
  2. 展示可编辑回执确认条。
  3. 用户确认后入账。
- 目标：3 步内完成。

### 3.2 模糊输入澄清
- 输入："刚才买东西"
- 处理策略：
  1. 尝试从上下文推断（常见分类、近期模式）。
  2. 无法确定时只问 1 个问题（金额或分类二选一）。
  3. 获取答案后直接回执确认。

### 3.3 错账修复
- 输入："不对，刚才是 30"
- 处理策略：
  1. 识别为“最近一笔修正意图”。
  2. 定位最近可编辑交易并更新。
  3. 返回修正后回执和撤销入口。

### 3.4 连续记账
- 触发：用户高频重复记录同类消费。
- 处理策略：
  1. 回执上显示“再记一笔”。
  2. 复用上次分类和备注，仅需改金额（或直接确认）。

### 3.5 预算关怀提醒
- 触发条件：明显偏离预算阈值时才提醒。
- 文案要求：
  - 不指责。
  - 只说 1 个事实 + 1 个可执行建议。
  - 示例：本周餐饮较上周增加 28%，是否把本周外卖预算上限临时设为 200 元？

## 4. 技术落点（与现有实现对齐）

- Prompt 策略：在系统提示词中明确“最小打扰澄清”和“可解释建议”规则。
- 解析契约：继续复用 `<DATA>...</DATA>` 结构化回执，避免前后端语义偏差。
- 数据层：
  - 交易主真相源仍为 Room。
  - AI 建议依赖本地统计结果，不阻塞记账主流程。
- UI 层：
  - `ChatScreen` 增加回执内联编辑入口。
  - `Ledger` 与 `Home` 统一展示修正后的 `recordTimestamp`。

## 5. 验收指标（建议）

- 首次记账完成率（无引导）：>= 90%
- 单轮记账成功率：>= 85%
- 纠错平均操作步数：<= 2
- 建议提醒关闭率（7 日）：<= 20%
- AI 相关负反馈率（“啰嗦/打扰/听不懂”）：连续两周下降

## 6. 迭代边界

- 当前不做：应用内 SpeechRecognizer Mic 输入。
- 当前不做：云同步正式上线（仅预研与 PoC）。
- 低优先级保留：CSV/JSON 导入导出作为备份迁移通道。

## 8. 范围冻结补充（2026-04-03）

后续 AI 相关需求收敛为以下四项：
- 一次性记录多笔账单（批量回执与逐笔落库）
- 时间语义简化（仅保留单时间字段，修改账单后更新时间）
- 从 Prompt 拼接演进到 Agent + 工具调用
- Android 16 Rich Ongoing Notifications（前台任务可视化）
