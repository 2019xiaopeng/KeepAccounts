# P3-S4 人设化本地路由优化方案（已落地）

- 日期：2026-04-09
- 状态：Implemented
- 目标：完成本地路由人设化、用户可见层零日志泄漏、update/query 口语稳定性增强
- 影响范围（已实施）：`ChatRepository`、`AgentStyleFormatter`、`ChatScreen`、query/update 规则与测试集

## 0. 本轮落地结果（2026-04-09）

1. 本地 write/query/stats 可见文案完成去技术化：
- 不再向用户直接展示“结构化结果/追踪ID/sampleSize”等调试字段。
- 查询和统计结果改为自然语言摘要，技术明细通过隐藏 NOTE payload 透传给 UI 卡片解析。

2. 本地回复节奏人设化：
- 本地命中路径增加轻量延迟窗口，避免“机械秒回”体感。
- `AgentStyleFormatter` 改为“确认动作 + 可理解结果 + 轻建议”风格。

3. update 口语可靠性增强：
- 明确写操作词（update/delete）优先于 query override。
- “最近一笔/最新一笔/刚刚那笔”等锚点优先定位目标记录。
- update 无显式时间线索时保持原交易时间戳，避免误改时间导致“最近一笔”排序漂移。

4. Chat 卡片可读性增强：
- `ChatScreen` 优先解析隐藏 `<NOTE>` payload 生成 insight 卡片。
- 保留 legacy 可见文本解析兜底，保证历史消息兼容。

5. 回归测试完成：
- 重点套件：`AgentStyleFormatterTest`、`ChatRepositoryBatchLedgerTest`、`ChatRepositoryTimeSemanticsTest` 通过。
- 全量单测：`:app:testDebugUnitTest` 通过。

## 1. 问题复盘（来自真机反馈）

1. 本地写入命中后回复“瞬时返回 + 结构化日志口吻”，缺少人设化交流节奏。
2. update 口语指令（如“帮我把最近一笔改成30块”）出现失败，暴露底层风格化日志，不符合用户话术。
3. 查询口语（如“统计过去一周花了多少钱”）存在命中不稳定，用户感知为“查不到/查不准”。

## 2. 当前实现观察（设计依据）

1. 本地 write 路由命中后由 `handleWriteIntent` 直接输出 `styleFormatter.formatWrite(...)`，文案默认包含：
- `结构化结果：...`
- `追踪ID：...`

2. 当前风格层 `AgentStyleFormatter` 的设计目标偏“可观测性可追踪”，不等于“用户对话体感”。

3. update 意图草稿构造与目标定位基于轻规则：
- 依赖金额/关键词抽取 + `scoreUpdateTarget` 打分
- 缺少“最近一笔/上一笔/刚刚那条”这类强锚点优先级策略
- 失败时直接展示偏技术风格结果

4. 时间窗解析虽然覆盖部分周窗口词，但缺少统一语义归一层与口语变体兜底策略，导致“用户表达多样性 > 规则覆盖”时命中不稳定。

## 3. 设计目标

1. 继续保持本地路由低延迟，但输出要“像真人管家”，而不是“工具日志”。
2. 用户可见层做到“零技术噪音”：不暴露结构化统计术语、追踪ID、调试字段。
3. update/query 口语命中率提升，失败时提供可执行下一步，而不是报错式文本。
4. 保留调试与追踪能力，但迁移到不可见通道（日志/内部字段）。

## 4. 核心方案（拟）

### 4.1 双通道响应模型（用户可见 vs 内部可观测）

1. 用户可见通道：只输出自然语言与回执卡片。
2. 内部可观测通道：结构化结果、requestId、tool stats 仅写入内部日志或隐藏字段。
3. 默认关闭“结构化结果/追踪ID”对用户展示；仅在调试模式可开启。

### 4.2 本地命中的人设化回复生成

1. 用“动作 + 场景 + 结果”生成自然语言：
- 动作：create/update/delete/query/stats
- 场景：从 category/remark/时间词推断（如打车、餐饮、医疗）
- 结果：成功/部分成功/失败

2. 文案分层：
- 首句：确认动作（例如“好嘞，这笔打车我记上了。”）
- 第二句：轻关怀（例如“这是回家还是出门玩呀？”）
- 第三句（可选）：下一步建议（仅失败/部分成功时）

3. 人设文案引擎优先用模板库（确定性、可测），而不是每次都请求大模型。

### 4.3 “有节奏但不慢”的返回策略

1. 本地命中时增加最小可感知停顿（例如 500~1200ms 抖动区间）。
2. 与现有 typing dots 配合：
- 请求处理中：仅显示 typing dots
- 完成后：整段自然语言 + 卡片一次出现

3. 目标是“像人在思考一下再回复”，不是“机械秒回日志”。

### 4.4 update 可靠性增强

1. 引入“更新目标锚点优先级”策略：
- 明确 transactionId > 明确时间/分类 > `最近一笔/最新一笔` > 默认最近记录

2. 对“改成30块”类语句构建显式 UpdateFrame：
- `targetSelector`（最近一笔/某天某类）
- `patch`（amount/category/date/...）
- `confidence`

3. 当 `confidence` 低于阈值时，不直接失败：
- 返回可执行澄清句（例如“我定位到最近一笔交通50元，要改成30元吗？”）

4. 失败文案去技术化：
- 用户面不出现工具字段和内部统计。

### 4.5 过去一周/近一周语义归一

1. 新增统一窗口归一层（WindowNormalizer）：
- “过去一周/最近一周/近7天/本周（滚动）”归一到同一窗口语义

2. 增加优先解析顺序：
- 显式天数 > 周/月自然语言 > 默认窗口

3. 对“统计过去一周花了多少钱”固定映射为：
- window=`last7days`
- metric=`total_amount`
- groupBy=`category`（或按需求聚合总额）

## 5. 与当前用户反馈的一一对应

1. 对“打车50瞬间日志回复”：
- 采用 4.1 + 4.2 + 4.3，去掉用户可见日志字段，保留真人口吻与可感知停顿。

2. 对“最近一笔改30失败”：
- 采用 4.4，提升 target 选择优先级与澄清策略，降低硬失败率。

3. 对“统计过去一周花了多少钱查不到”：
- 采用 4.5，做口语时间窗统一归一与 query 参数稳定化。

## 6. 分阶段落地建议（已执行）

1. Phase A：响应通道分层 + 文案模板引擎 + 停顿策略。
2. Phase B：update 的 UpdateFrame 与锚点优先级。
3. Phase C：时间窗归一层 + query/stats 口语回归。
4. Phase D：灰度开关与指标观测（命中率、失败率、用户纠错率）。

## 7. 测试与验收结果

1. 本地 write 人设回归：
- 输入“打车50”应返回自然口吻，不出现 `结构化结果`、`追踪ID`、JSON。

2. update 回归：
- 输入“把最近一笔改成30块”应命中最近记录并成功修改。
- 低置信度场景应触发澄清而非直接失败。

3. query/stats 回归：
- “统计过去一周花了多少钱”稳定命中 last7days，并返回可理解结果。

4. 非功能回归：
- 保持现有速度目标，不引入明显卡顿。
- 调试信息在内部可追踪但用户面不可见。

## 8. 待决策项

2. 人设文案是否按用户配置（风格/亲密度）可调。
3. 低置信度 update 是否强制二次确认。
4. 调试信息查看入口放在开发者模式还是独立调试页。
