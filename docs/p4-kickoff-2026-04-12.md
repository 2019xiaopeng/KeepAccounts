# Phase4 启动文档（2026-04-12）

- 阶段：Phase4（P4）
- 分支建议：`feat/p4-rich-ongoing-notify`
- 结论：Go（可启动）
- 前置基线：Phase3 已收尾并进入观测维护期

## 1. 启动目标

1. 完成 Android 16 Rich Ongoing Notifications 与低版本降级方案。
2. 打通“Agent 执行 -> 通知状态 -> 执行详情页 -> 聊天结果卡片”闭环。
3. 保持与 Phase3 新能力一致：
   - 工具编排与双模型路由结果可追溯。
   - 对话体验升级后的时间语义与状态表达一致。

## 2. 范围定义

### 2.1 In Scope

1. 前台服务承载 AI 执行过程。
2. 通知状态机：`PROCESSING`、`PARTIAL_SUCCESS`、`SUCCESS`、`FAILED`。
3. 通知点击跳转执行详情页。
4. Android 16 Rich Ongoing Notifications 实现。
5. Android 16 以下普通前台通知降级实现。
6. 执行详情页展示关键观测字段：
   - `plannerModelUsed`
   - `chatModelUsed`
   - `routeReason`
   - `escalatedToPro`
   - 结果摘要（成功/失败条目）
7. 文案口径统一：通知、详情页、聊天卡片三处一致。

### 2.2 Out of Scope

1. 新增云同步与多端消息推送。
2. 账号体系与登录绑定能力。
3. 改变 Phase3 已定的工具协议与风险门禁策略。

## 3. 验收标准

1. 功能验收
   - 批量记账进入后台后通知持续可见。
   - 处理中断网恢复后状态正确收敛。
   - 点击通知可进入详情页并看到完整执行摘要。
2. 一致性验收
   - 详情页与聊天卡片的成功/失败计数一致。
   - 时间语义文案与聊天页一致，不出现相互矛盾描述。
3. 兼容性验收
   - Android 16：Rich Ongoing Notifications 正常。
   - Android 16 以下：降级通知路径正常。
4. 质量验收
   - `:app:compileDebugKotlin` 通过。
   - `:app:testDebugUnitTest` 通过。

## 4. 风险与应对

1. 风险：通知状态与聊天结果不一致。
   - 应对：以统一执行结果源生成通知文案与详情页摘要。
2. 风险：高低版本行为分叉导致回归遗漏。
   - 应对：为 Android 16 增强路径和低版本降级路径分别建立验收清单。
3. 风险：观测字段未透传或丢失。
   - 应对：在详情页入口加字段完整性校验与默认降级文案。
4. 风险：后台恢复过程中状态机重复推进。
   - 应对：引入 requestId 幂等检查，避免重复写入终态。

## 5. 里程碑

1. M0（D0）：启动评审完成，范围冻结。
2. M1（D1-D2）：前台服务 + 通知状态机实现。
3. M2（D3-D4）：Android 16 富通知与低版本降级打通。
4. M3（D5）：执行详情页联动与观测字段展示完成。
5. M4（D6）：全量回归、文档收口、提交 PR。

## 6. 与 Phase3 的衔接要求

1. 不回退 Phase3 的工具边界与安全策略。
2. 保留并复用已有路由观测字段，不新增破坏性协议变更。
3. 与 P3-S6 对话体验规范保持一致：
   - 聊天记录删除仅影响 `chat_messages`。
   - 时间分割语义与通知时间表达一致。
