# ADR 030: 聊天气泡时间戳与回执记录时间精度修复

- 日期: 2026-03-29
- 状态: Accepted

## 背景

聊天页此前仅展示消息气泡，不展示每条消息的发送时间；同时 AI 回执链路在解析 date 字段时只保留到日期，导致入库记录时间常退化为 00:00。

该问题会带来两类体验偏差：

1. 聊天上下文缺乏时间感，用户无法快速判断消息先后。
2. 账单真实发生时分丢失，账本、搜索、回执卡片间时间语义不一致。

## 决策

1. 维持交易实体使用完整时间戳字段（recordTimestamp: Long），不降级为仅日期类型。
2. 在 AI System Prompt 中注入“当前系统准确时间（精确到分钟）”，并强制要求 <DATA> JSON 输出 recordTime（yyyy-MM-dd HH:mm）。
3. Repository 入账时间解析采用分层优先级：
   - 优先解析用户原话中的显式日期/时分（如“昨晚10点”）。
   - 次优先解析 AI 返回的 recordTime。
   - 再回退解析 date（仅日期时补当前时分）。
   - 最终回退当前系统时间。
4. 聊天 UI 在每条消息气泡下新增低干扰时间戳（HH:mm），并保持方向一致：用户右对齐、AI 左对齐。
5. 回执卡片“记录时间”显示改为交易记录时间，不再使用聊天消息创建时间。

## 影响

- 聊天界面的消息时间可见性提升，阅读顺序更清晰。
- 回执、账本、搜索中的记录时分更一致，00:00 默认时间显著减少。
- 对历史仅含 date 的旧回执，仍能兼容解析并以当前时分补全显示。

## 代价与权衡

- 时间解析逻辑复杂度上升（需处理中文相对时间与时段词）。
- 若用户语句缺乏明确时分，按“当前系统时间”补全会引入近似值；但该策略优于统一落到 00:00。
- 选择 Long 时间戳而非 LocalDateTime，可继续复用现有 Room 结构并避免迁移成本。
