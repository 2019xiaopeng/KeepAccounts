# ADR-044: Phase3-S4 对话完成态呈现与零 JSON 可见策略

- 状态：Accepted
- 日期：2026-04-09
- 决策者：KeepAccounts 团队
- 关联阶段：Phase3 Step4 (P3-S4)
- 前置 ADR：ADR-042、ADR-043

## 背景

在 P3 Agent 化后，聊天层仍有两类体验与安全风险：

1. 助手回复采用流式增量落库，导致“气泡先出现、文本再逐步填充”，与目标体验“输入中三点动画 -> 完整气泡瞬时出现”不一致。
2. 虽然已剥离 `<DATA>/<RECEIPT>` 等标签 payload，但模型在工具调用场景下仍可能直接输出无标签 JSON（含 Markdown 代码块、内联对象或截断片段），存在被用户看到的风险。

## 决策

1. 回复写库策略改为“完成态写入”：
- 保持网关流式读取能力以兼容现有链路。
- 取消 TextDelta 阶段的增量写库与分段揭示。
- 仅在流结束后一次性写入最终气泡内容。

2. 可见文本执行“零 JSON 可见”清洗策略：
- 先剥离 `<DATA>/<RECEIPT>/<NOTE>/<THINK>` payload。
- 再剥离无标签 JSON 风险载体：
  - 任意 Markdown 代码块与未闭合代码块片段。
  - 平衡大括号/方括号 JSON 段（严格 JSON 与宽松 JSON 线索）。
  - 尾部未闭合 JSON 片段。
  - 残留的 JSON 键值行与孤立符号行。

3. UI 层保留同等级兜底清洗：
- 对历史消息或异常数据源，继续在 `ChatScreen` 做同口径可见文本清洗，避免单点失效。

## 备选方案与否决原因

1. 仅将请求参数改为 `stream=false`。
- 否决：改动网关协议面更大，且会降低后续需要流式解析能力的灵活性。

2. 只在 Repository 清洗，不在 UI 兜底。
- 否决：一旦历史数据或其他写入路径带入 JSON，UI 仍可能暴露。

3. 仅靠标签规则（`<DATA>/<RECEIPT>`）清洗。
- 否决：无法覆盖无标签 JSON 幻觉与截断输出。

## 影响

正向：

1. 聊天视觉节奏更接近主流 IM：等待阶段只显示 typing dots，完成后整段出现。
2. JSON 泄漏风险显著下降，用户仅看到自然语言与结构化回执卡片。
3. 回执解析仍可使用隐藏 payload，不影响记账卡片与事务落库。

成本：

1. 失去流式“字字出现”的即时反馈。
2. 文本清洗规则复杂度提高，需要持续回归验证。

## 验证

1. 定向回归通过：
- `ChatRepositoryBatchLedgerTest`
- `ChatRepositoryTimeSemanticsTest`

2. 全量单测通过：
- `:app:testDebugUnitTest`

3. 新增关键断言：
- 流式 delta 场景仅在完成后落库助手气泡（无中间 update）。
- 无标签内联 JSON 可解析并记账，但不可见文本不暴露 JSON。
- 未闭合 JSON 片段不会泄漏到聊天气泡。

## 后续

1. 增加 UI 自动化回放，验证“typing dots -> 完整气泡出现”的时序体验。
2. 对清洗规则加入属性测试，覆盖更多畸形 JSON 片段输入。
