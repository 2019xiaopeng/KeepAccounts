# ADR 025: 聊天非阻塞发送与日期可靠性修复

- Status: Accepted
- Date: 2026-03-29

## Context
发布前回归暴露三个关键问题：
1. 发送消息后聊天界面卡顿，直到整段回复完成才整体刷新。
2. 用户输入相对日期（例如“昨天买药 35”）时，记账日期仍被写成今天。
3. 手动记账页面日期不可编辑，保存时间被固定为当前时间。

## Decision
1. 聊天发送链路改为 IO 执行：
   - `ChatViewModel.sendMessage` 与 `deleteMessage` 使用 `Dispatchers.IO` 启动。
   - 避免网络流读取、文本分段与持久化占用 UI 线程，确保输入与流式气泡及时更新。
2. 记账日期采用“用户输入优先”的本地解析策略：
   - 在 `ChatRepository` 中优先解析用户原文日期（绝对日期 + 相对日期：昨天/前天/大前天/今天/明天）。
   - 若无法从用户输入解析，再回退到模型返回的 `date`。
   - 同步增强 system prompt：遇到相对日期必须换算为具体 `yyyy-MM-dd`。
3. 手动记账页面增加日期可编辑能力：
   - 新增日期选择器（DatePickerDialog）。
   - 保存时使用用户选定日期，不再固定写入当前时间。

## Why Not
- 为什么不只靠模型自己理解“昨天”：
  - 模型可能输出默认日期或上下文漂移，业务侧需本地兜底确保数据准确性。
- 为什么不继续在主线程执行发送：
  - 任意网络波动都可能直接卡住 UI，影响发送反馈与流式体验。

## Consequences
- 正向：
  - 聊天发送后用户消息可立即显示，流式更新不中断。
  - 相对日期记账准确性显著提升。
  - 手动记账支持回填历史日期，满足真实记账场景。
- 代价：
  - 日期解析逻辑增加，需要持续维护更多口语时间表达。

## Validation
- `:app:compileDebugKotlin` 通过。
- `:app:assembleDebug` 通过。

## Next
1. 可扩展“上周一/上个月 3 号”等相对时间短语解析。
2. 可加入 UI 自动化用例覆盖“发送后 1 秒内出现用户消息”与“昨天语句入账日期”断言。
