# ADR-045: Phase3-S4 本地命中人设化回复与可观测信息边界策略

- 状态：Accepted
- 日期：2026-04-09
- 决策者：KeepAccounts 团队
- 关联阶段：Phase3 Step4 (P3-S4)
- 前置 ADR：ADR-042、ADR-043、ADR-044

## 背景

当前本地路由在 write/query/stats 命中时，强调“结构化可追踪输出”，带来了三个体验问题：

1. 用户可见回复过于技术化（结构化结果、追踪ID等）。
2. 本地路径响应虽快，但缺少“真人管家”语气与互动感。
3. 口语 update/query 的命中与恢复策略不够稳健，失败时文案不可用。

## 决策

1. 建立“用户可见通道”和“内部可观测通道”边界：
- 用户可见：仅自然语言 + 回执卡片。
- 内部可观测：结构化结果、traceId、工具细节只进入日志/内部字段。

2. 本地命中回复采用人设模板引擎：
- 不再直接把结构化摘要当用户文案。
- 按动作/场景/结果生成自然语言三段式回复（确认、关怀、建议）。

3. 引入最小可感知停顿策略：
- 保持低延迟前提下，避免“瞬时机械回包”体验。

4. update/query 口语策略强化：
- update 使用目标锚点优先级与低置信度澄清策略。
- query/stats 使用统一时间窗归一层覆盖“过去一周”等口语表达。

## 备选方案与否决原因

1. 保持现状（结构化日志直接上屏）。
- 否决：与人设目标冲突，用户感知为“工具在说话”。

2. 本地命中一律转远端 LLM 生成话术。
- 否决：增加延迟与失败面，不符合“本地快速闭环”目标。

3. 只优化文案，不做可观测边界分层。
- 否决：调试字段仍可能污染用户可见层。

## 影响

正向：

1. 用户体验从“日志式回包”转向“管家式对话”。
2. 调试能力保留，同时避免技术字段外泄。
3. update/query 口语命中稳定性与失败恢复体验提升。

成本：

1. 文案模板与意图归一规则需要维护。
2. 需要新增回归测试矩阵与灰度开关。

## 实施状态

1. 当前状态：Implemented（2026-04-09）
2. 关键落地点：
- `AgentStyleFormatter`：用户可见 write/fallback 文案去结构化，改为自然对话风格。
- `ChatRepository`：
	- query/stats 可见文案去日志化，内部通过隐藏 `<NOTE>` 保留解析数据。
	- 本地命中增加轻量延迟，优化“秒回像日志”体感。
	- update/query 路由冲突处理中，显式写操作优先。
	- update 无时间线索时保留原 `recordTimestamp`，避免记录重排。
- `ChatScreen`：优先从隐藏 NOTE payload 解析 insight 卡片，fallback 到 legacy 解析。
3. 已验证结果：
- 重点测试套件通过（`AgentStyleFormatterTest`、`ChatRepositoryBatchLedgerTest`、`ChatRepositoryTimeSemanticsTest`）。
- 全量单测 `:app:testDebugUnitTest` 通过。
4. 补充落地点（2026-04-09）：
- 查询语义补强：支持“过去一周花了多少钱”等自然表达直达本地 stats。
- 分类语义补强：新增“吃饭/吃中饭/中饭/夜宵”等口语映射到“餐饮美食”。
- 回复切分修正：单段回复默认单气泡，避免句中硬切。
- UI 去冗余与情绪化：失败提示去重 + 关键信息点 Emoji 化。

## 实施前置与验证

1. 先完成方案落地设计文档：
- `docs/p3-s4-humanized-local-routing-optimization-plan.md`

2. 实施后验收（本 ADR 对应）应覆盖：
- 本地 write 人设回复与停顿行为
- update “最近一笔改30块”命中
- query “统计过去一周花了多少钱”稳定命中
- 用户可见层不出现结构化调试字段

## 结论

采用“本地执行快速闭环 + 用户文案人设化 + 可观测信息内收”的混合策略，在不牺牲速度的前提下提升可用性与可信度。
